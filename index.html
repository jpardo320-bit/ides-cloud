The problem

Every encryption vendor sees your plaintext.

AWS KMS, Azure Key Vault, HashiCorp Vault, every password manager — they all process plaintext on their hardware at some point. Their security posture is built on "we won't look." IDES is built so we can't.

Architecture

How the system is put together.

Plaintext and keys stay inside your environment. Only ciphertext crosses to IDES Cloud — where the gateway, key daemon, audit catalog, and storage all operate on data they cannot decrypt.

The trust boundary is the green box. Everything IDES Cloud runs — gateway, key daemon, audit catalog, storage — sits on the other side of it, operating on ciphertext alone.

Four tiers, one login

The same login key unlocks all four encryption tiers. Each tier pays only the per-record cost it actually needs — no KMS round-trip per row.

Tier 1 · Vault

Master secrets

Per-payload AES-256-GCM with BLAKE3-keyed isolation. Leak one entry, leak only that entry.

Tier 2 · Document

Files & objects

S3-compatible gateway with end-to-end streaming encryption. RSS bounded at ~16-32 MB for any object size.

Tier 3 · Record

Database rows

Per-record HKDF AES-256-GCM with split-secret pepper. ~10¹¹⁵ combinations per record.

Tier 4 · Stream

Logs & telemetry

XChaCha20-Poly1305 with sequence-bound nonces. Replay, reorder, truncation detected at decrypt.

Defense in depth

The math behind "we can't decrypt it."

IDES layers a geometric key-derivation stage on top of standard credential hashing. After a full database leak, an attacker still faces a brute-force search space measured in hundreds of orders of magnitude.

The geometric key-derivation layer — three nested layers of bézier paths. Zoom through them and the keyspace compounds.

3.7 × 10²²⁴
Total combinations per account — credential layer and geometric layer combined. The observable universe holds roughly 10⁸⁰ atoms.
~10¹⁹⁵×
More combinations than any password-hashing peer at typical password strength.
6.4 × 10¹⁴⁵
Extra combinations contributed by the geometric layer alone, on top of the credential layer.
~10¹¹⁵
Combinations per database record (Tier 3) and per stream (Tier 4), from the 256-bit server-side pepper.
1.9 × 10¹¹²
Per-account search space remaining even against a quantum adversary under Grover's algorithm.
Standard primitives
AES-256-GCM, HKDF-SHA-256, Ed25519, BLAKE3, XChaCha20-Poly1305 — published, reviewable, no proprietary cryptography.

Keyspace figures are calculated from the IDES key-derivation design. They describe brute-force resistance, not a guarantee against implementation flaws — which is why the core engine is open source and built only on standard primitives.

Benchmarks

The architectural advantage, in numbers.

KMS-style services make a network API call for every operation. IDES keeps the session key in process — there's no network on the per-record hot path. That's the difference between millisecond and microsecond latency.

System Per-operation model Hot-path network call Typical per-op latency
IDES Session key in process No 11–37 µs
AWS KMS Network API call per operation Yes ~ms per call
HashiCorp Vault Network API call per operation Yes ~ms per call
Azure Key Vault Network API call per operation Yes ~ms per call

The per-record gap is roughly two to three orders of magnitude. For workloads that encrypt every database row or every log line, the difference between in-process and round-trip is the difference between viable and not.

FAQ

What people ask first.

How is this different from AWS KMS?
AWS KMS handles keys; you still encrypt and decrypt your data inside an AWS-owned process, which means AWS sees the plaintext at the moment of crypto operation. IDES Cloud never touches plaintext — the SDK in your environment encrypts before anything leaves your boundary. Also: KMS round-trips cost ~1ms each, untenable for high-volume per-record workloads. IDES does per-record encryption in-process at microsecond latency.
Do you ever see our data?
No — by architecture. Customer plaintext is encrypted client-side using a key derived from your KMS. The ciphertext we hold is opaque to us. If subpoenaed, we can produce ciphertext + audit catalog metadata; we cannot produce plaintext because we don't have the keys.

Can I self-host instead?
Yes — IDES is also available as software you run yourself. Tier 3 BYOK deployments install in your own AWS/GCP account; you control the entire infrastructure. Many customers run a mix: self-hosted Tier 3 for production data, IDES Cloud for dev/staging and ancillary use cases.
What if you go out of business?
Source code is licensed under MIT/Apache-2.0. You'd have everything to continue running the software yourself. Customer-controlled keys also mean your data is recoverable independent of us — your KMS still works, your ciphertext is still readable with the keys you control.
